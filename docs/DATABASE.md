# Database

As-built reference. The full schema lives in [`backend/database/schema.sql`](../backend/database/schema.sql) (generated from the migration files in `backend/database/migrations/`, applied in numeric order) with seed data in `backend/database/seeders/`. This document explains the *shape* of the design; read the schema file for exact column types/constraints.

42 tables, MySQL 8, InnoDB, `utf8mb4_unicode_ci`. No ORM — every query is hand-written and parameterized.

## Conventions used throughout

- **Soft delete**: tables with a real lifecycle (`products`, `categories`, `brands`, `customers`, `suppliers`, `expense_categories`, `expenses`) use `deleted_at DATETIME NULL` rather than a hard `DELETE`. Every read query filters `WHERE deleted_at IS NULL`. Append-only tables (`inventory_movements`, `activity_logs`) have no delete path at all, by design.
- **Audit columns**: `created_by`/`updated_by` (nullable `FK → users.id`, `ON DELETE SET NULL` — a deleted user doesn't cascade-delete their history) alongside `created_at`/`updated_at` on every mutable table.
- **Status enums over boolean flags**: `status ENUM('active','inactive')` (or module-specific states like `sales.status ENUM('completed','voided')`) rather than a bare `is_active` boolean — leaves room for future states without a schema migration.
- **Money as `DECIMAL(14,2)`**, never `FLOAT`/`DOUBLE` — exact arithmetic for currency.
- **Document numbering**: every business document (`purchase_orders.purchase_number`, `sales.sale_number`, `returns.return_number`, `stock_transfer_requests.transfer_number`, `customers.customer_code`, `products.code`) is generated by `sequence.repository.generateCode()` against `document_sequences`, keyed by `(document_type, year)` — resets numbering each calendar year, collision-free under concurrency (see [ARCHITECTURE.md](ARCHITECTURE.md)).

## The inventory ledger

`inventory` holds current quantity per `(product_id, branch_id)`. `inventory_movements` is the append-only, complete history of every change to that quantity — `previous_stock` and `new_stock` are recorded on every row, so the ledger is self-verifying (each row's `previous_stock` should equal the prior row's `new_stock` for that product/branch). Every write to `inventory_movements` goes through `inventory.repository.recordMovement()`; nothing else ever inserts into that table. `movement_type` is one of `purchase`, `sale`, `return`, `transfer_out`, `transfer_in`, `adjustment`, `opening_balance`, `manual_correction`.

`inventory_adjustments` is a thin side-table that adds a `reason`/`description` to a manual correction — it references the `inventory_movements` row it explains rather than duplicating the movement itself.

## Two-step approval workflows

`stock_transfer_requests` and `returns` share the same shape: a `status ENUM` starting at `pending`, moved to `approved` or `rejected` by a second actor (`approved_by`/`approved_at`), with inventory impact deferred until approval — never at request creation. This mirrors real business process (a request and its approval are genuinely two separate events) and means creating a request never risks a partial or incorrect stock change.

## Reference tables intentionally left thin

- **`carwash_services`**: 4 rows seeded once (`Normal Wash`, `Full Wash`, `Engine Wash`, `Interior Cleaning`), no management UI — the spec's Car Wash feature list didn't call for managing services, only picking from them.
- **`expense_categories`**: 9 rows seeded once (`Rent`, `Electricity`, `Water`, `Fuel`, `Salary`, `Maintenance`, `Transport`, `Office Supplies`, `Other`), same reasoning.

## Tables with no application code writing to them yet

- **`audit_logs`**: exists in the schema since Phase 0 but no phase has written to it — every module uses `activity_logs` instead (a human-readable timeline entry per significant action: "Sale SAL-2026-000001 completed", "Purchase PUR-2026-000001 received from..."). `audit_logs` was intended for a lower-level field-by-field change record; nothing in the spec's per-phase feature lists asked for one, so it stays reserved rather than populated with a design nobody's validated yet.
- **`system_backups`**: written to by both the manual backup trigger (Settings → Backups → "Run Backup Now") and the scheduled daily job (`backend/jobs/backupJob.js`, added in Phase 24) — `trigger_type` distinguishes the two.

## Branch scoping

Every table that represents branch-owned data has a `branch_id` column (`sales`, `purchase_orders`, `expenses`, `carwash_transactions`, `inventory`, etc.) or reaches one through a join (`inventory_movements` via `branch_id` directly; `returns`/`stock_transfer_items` via their parent sale/transfer). `branchScope.getAccessibleBranchIds(user)` is the single function every service calls to translate "what can this user see" into a SQL filter — see [ARCHITECTURE.md](ARCHITECTURE.md) for how it's used.

## ER overview (textual)

```
users --(role_id)--> roles --(role_permissions)--> permissions
users --(branch_id)--> branches <--(user_branches, many-to-many)-- users

products --(category_id)--> categories
products --(brand_id)--> brands
products <--(product_id, branch_id)--> inventory --> inventory_movements

purchase_orders --(supplier_id)--> suppliers
purchase_orders --> purchase_items --(product_id)--> products
                 \-> triggers inventory_movements (movement_type='purchase')

sales --(customer_id, nullable)--> customers
sales --(cashier_id)--> users
sales --> sale_items --(product_id)--> products
       \-> triggers inventory_movements (movement_type='sale')
sales --> sale_payments (1-to-many, mixed payment support)

returns --(sale_id)--> sales
returns --> return_items --(sale_item_id)--> sale_items
        \-> on approval, triggers inventory_movements (movement_type='return')

stock_transfer_requests --(source_branch_id, destination_branch_id)--> branches (×2)
stock_transfer_requests --> stock_transfer_items --(product_id)--> products
                         \-> on approval, triggers inventory_movements ×2 per item
                            (movement_type='transfer_out' at source, 'transfer_in' at destination)
```
